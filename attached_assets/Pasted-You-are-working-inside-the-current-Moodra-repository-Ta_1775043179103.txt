You are working inside the current Moodra repository.

Task:
Implement a robust two-phase export pipeline for books with designer pages, especially to avoid failures in Cyrillic export flow when heavy full-page images are present.

Context:

* Regular export of text/book blocks works.
* Export often breaks when designer pages are included.
* Designer pages are already uploaded separately and stored as files, while book content is exported correctly on its own.
* We must keep the current successful core export logic for text/content blocks.
* We must stop embedding designer pages into the same heavy rendering path too early.
* Goal: export the core book first, then inject designer pages into the final output as a second step.

What to implement:

1. Introduce a two-phase export architecture:
   Phase A: Core export

   * Export only book content, chapters, blocks, typography, language logic, Cyrillic hyphenation, page settings, margins, page numbering, etc.
   * Do NOT inline designer pages during the main content rendering phase.
   * Ignore designer pages temporarily during core export, but preserve their intended placement metadata.

   Phase B: Post-processing / injection

   * After the core file is generated successfully, run a second pipeline step that inserts designer pages into the final file at the exact configured positions.
   * Treat designer pages as standalone full-page assets.
   * Insert them by file path / URL reference from storage, not by bloating the main content tree with base64 blobs early.

2. Create a normalized placement model for designer pages:
   Each designer page must have explicit placement metadata such as:

   * id
   * imageUrl or filePath
   * placementType: "before_chapter" | "after_chapter" | "before_block" | "after_block" | "absolute_index"
   * chapterId / blockId if relevant
   * absoluteIndex if relevant
   * fitMode: "cover" | "contain"
   * pageBreakBefore: boolean
   * pageBreakAfter: boolean
   * backgroundColor optional
   * enabled: boolean

3. Build an export manifest before generation:
   Create a manifest that contains:

   * ordered core content items
   * ordered designer page insertions
   * final intended sequence of pages/sections
     This manifest should be serializable and debuggable.

4. Refactor export logic:
   Extract export-related logic from the current large route handler into dedicated services:

   * export/buildExportManifest.ts
   * export/exportCoreDocument.ts
   * export/injectDesignerPages.ts
   * export/types.ts
   * export/utils.ts

5. Implement safe injection strategy:
   If export target is DOCX:

   * inject designer pages as separate full-page sections using image runs / full-page image sections
   * scale images to page size according to current page settings
   * respect A4/A5 and margins rules
   * do not distort aspect ratio unless fitMode = "cover"

   If export target is PDF:

   * generate the normal PDF first
   * reopen/patch it in a second pass
   * insert dedicated image pages at the required indexes
   * do not rerender the whole book from scratch if not needed

6. Very important memory/performance rules:

   * never keep all heavy designer page images as base64 in one giant in-memory structure if avoidable
   * stream or load them one by one during injection
   * use file paths or buffers lazily
   * free intermediate buffers where possible
   * log memory usage around export start / core export done / injection done

7. Cyrillic safety:

   * preserve the existing Cyrillic-specific text logic for RU/UK/UA content
   * designer pages must not go through the same text/hyphenation flow
   * text rendering and designer page insertion must become decoupled
   * no regression for Latin exports

8. Fallback behavior:

   * if designer page injection fails, return a clear structured export error
   * optionally provide a fallback export without designer pages, but only if explicitly requested by a flag
   * never silently corrupt the output

9. Add diagnostics:

   * logs for total number of designer pages
   * each page file size
   * insertion positions
   * export phase durations
   * whether failure happened in core export or post-processing

10. Deliverables:

* implement the code changes
* explain which files were changed
* explain the new architecture
* include a short migration note if any existing data model changed
* include sample test cases:
  a) Latin book without designer pages
  b) Latin book with designer pages
  c) Cyrillic book without designer pages
  d) Cyrillic book with multiple heavy designer pages
* clearly mark any TODOs if some edge cases remain

Important:

* Do not rewrite the whole app.
* Keep current working book export behavior intact.
* Focus on minimal invasive refactor with strong reliability.
* Prefer practical, production-friendly code over theoretical perfection.
